# <a name="concepts-unique-to-project-based-quotes"></a>Концепції, що є унікальними для цінових пропозицій на основі проекту

_**Застосовується до:** Project Operations для сценаріїв на основі ресурсів і відсутності запасів, полегшене розгортання: угоди та виставлення рахунків-проформ_

У програмі Dynamics 365 Project Operations є два типи цінових пропозицій: цінові пропозиції за проектами та цінові пропозиції збуту. Ці два типи цінових пропозицій мають перелічені нижче відмінності.

- **Сітки для елементів рядків**: у ціновій пропозиції збуту існує лише одна сітка для елементів рядків. У проектній ціновій пропозиції існує дві сітки для елементів рядків. Одна сітка призначена для проектних рядків, а інша — для рядків продуктів.
- **Активація та виправлення**: цінові пропозиції збуту підтримують активацію та виправлення. Ці процеси не підтримуються для проектних цінових пропозицій.
- **Вкладені замовлення**: до цінової пропозиції збуту можна вкладати кілька замовлень. До проектної цінової пропозиції можна вкласти лише один сервісний договір проекту.
- **Виграш цінової пропозиції**: коли ви виграєте цінову пропозицію збуту, пов'язана потенційна угода може залишатися відкритою. Після того, як цінову пропозицію проекту буде виграно, закриється відповідна потенційна угода.
- **Поля та концепції**: цінова пропозиція збуту не містить деяких полів і ключових понять, присутніх у проектній ціновій пропозиції. Поля включають **Договірну одиницю**, **Диспетчер облікових записів** і **Рахунок на ім'я контактної особи**.  
- **Тип**: цінові пропозиції збуту та проектні цінові пропозиції також визначаються полем, яке базується на наборі параметрів і має назву **Тип**. Для цінової пропозиції збуту це поле має значення **На основі позицій**. Для цінової пропозиції проекту воно має значення **На основі роботи**.

Цей розділ зосереджується на проектних цінових пропозиціях.

Проектна цінова пропозиція у програмі Project Operations може мати кілька позицій або рядків цінової пропозиції. Однак проектна цінова пропозиція має дві сітки для елементів лінійки. Одна сітка призначена для проектних ліній, які дають змогу отримати докладні прогнози. Інша сітка призначена для лінійок на основі продуктів, які використовують просту ціну за одиницю та принцип залежності від кількості.

- **На основі проекту**: пропонована вартість визначається після оцінки обсягу необхідних робіт. Ви можете поверхнево оцінити роботу безпосередньо у подробицях під кожним рядком цінової пропозиції, або ж на основі кошторисних оцінок, використовуючи проект та план проекту. Рядки проектної цінової пропозиції можна знайти лише в проектних цінових пропозиціях, створених за допомогою засобу Project Operations. Цей тип лінії цінової пропозиції – це настроювана форма доданих ліній цінової пропозиції, яка доступна у Microsoft Dynamics 365 Sales.

- **На основі продукту**: пропонована вартість визначається на підставі кількості проданих одиниць, а також вартості збуту за одиницю продукту. Продукт у продуктовій лінійці може походити з каталогу продуктів у Sales, або це може бути продукт, який ви визначаєте. Такий тип рядків цінової пропозиції також доступний у проектних цінових пропозиціях, створених за допомогою Project Operations.

Сума цінової пропозиції — це загальний результат, отриманий від лінійки на основі продуктів та проектних ліній.

> [!NOTE]
> Цінові пропозиції та рядки цінових пропозиції не потрібні для програми Project Operations. Процес проекту можна почати з сервісного договору проекту (проданого проекту). Проте потенційні угоди завжди потрібні, незалежно від того, чи ви починаєте із цінової пропозиції, чи з сервісного договору проекту.

## <a name="project-based-quote-lines"></a>Позиції на основі проектів у ціновій пропозиції

Рядок цінової пропозиції на основі проекту в Project Operations має такі методи виставлення рахунків:

- Час і матеріал
- Фіксована ціна

### <a name="time-and-material"></a>Час і матеріал

Метод часу та матеріалів, який використовується для виставлення рахунків, залежить від споживання. Якщо вибрати цей метод виставлення рахунка, клієнт отримає рахунок відповідно до витрат за цим проектом. Рахунки-фактури створюються регулярно на основі дати. Під час процесу збуту пропонована ціна компоненту часу та матеріалу, забезпечує лише прогноз остаточної вартості клієнту. Продавець не зобов’язаний робити все можливе, щоб завершити проект з точно запропонованим значенням. Компоненти часу та матеріалу збільшують ризик клієнта. Клієнти можуть домовитися про додаткові умови, щоб не перевищувати певний поріг і мінімізувати ризик. У Project Operations не підтримується встановлення умов не перевищення вартості.

### <a name="fixed-price"></a>Фіксована ціна

У разі фіксованого методу розрахунків, продавець зобов'язується сам реалізувати проект для клієнта за фіксованою вартістю. Клієнту виставляється рахунок із фіксованим значенням лінійки пропозиції із фіксованою ціною, незалежно від витрат, які продавець бере на себе, щоб забезпечити виконання цієї лінійки. Вартість пропонованої лінійки з фіксованою ціною виставляється в рахунку одним із зазначених нижче способів. 

- Як загальна сума на початку або наприкінці проекту, або після досягнення проміжного етапу проекту. 
- Частковими рівними платежами з частотою на основі дат із фіксованим значенням в рядку цінової пропозиції. Ці часткові платежі відомі періодичні проміжні етапи.
- В часткових платежах з грошовим значенням, яке закривається по ходу виконання робіт або по досягненні конкретних проміжних етапів у проекті. У цьому разі значення кожного платежу може відрізнятися, але всі вони мають бути додані до фіксованого значення в рядку цінової пропозиції.

Програма Project Operations підтримує всі три типи розкладів рахунків-фактур для позицій з фіксованою ціною.

## <a name="transaction-classification"></a>Класифікація транзакцій

Організації, які надають професійні послуги, зазвичай надають цінову пропозицію та виставляють рахунки клієнтам за класифікацією витрат. Витрати представлені наступними класифікаціями транзакцій.

- **Час**: ця класифікація представляє вартість часу праці або людських ресурсів над проектом.
- **Витрати**: ця класифікація представляє всі інші види витрат за проектом. Оскільки витрати можна детально класифікувати, більшість організацій створюють підкатегорії, наприклад, подорожі, оренду автомобілів, готелі або канцелярські приладдя.
- **Комісія**: ця класифікація представляє різні накладні витрати, штрафи та інші витрати, які стягуються з клієнта. 
- **Податок**: ця класифікація представляє податкові суми, які користувачі додають під час введення витрат.
- **Матеріальна транзакція**: ця класифікація представляє фактичні дані щодо позицій продуктів на підтвердженому рахунку проекту.
- **Проміжний етап**: ця класифікація використовується для виставлення рахунків з логікою фіксованої ціни.

З кожним рядком в ціновій пропозиції може бути пов’язана одна або кілька класифікацій транзакцій. Після того, як цінову пропозицію буде виграно, зіставлення між класифікацією транзакцій і ціновою лінією буде передано сервісну роботу за договором.
  
Наприклад, цінова пропозиція може містити такі дві пропоновані лінії: 

- Консультаційна робота, в якій використовуються метод розрахунку часу та матеріалу, де застосовуються класифікації транзакцій часу і комісій. Наприклад, усі часові та комісійні транзакції для проекту, наприклад, **Впровадження AX Dynamics** виставляються в рахунку клієнту, залежно від часу та матеріалів, які використовуються. 
- Пов'язані транспортні витрати, які використовують метод фіксованого ціноутворення. Наприклад, усі витрати на відрядження за проектом, наприклад, **Впровадження AX Dynamics**, виставляються в рахунку за фіксованим грошовим значенням.

> [!NOTE]
> Комбінація проекту та класифікації транзакцій **Час**, **Витрати** і **Комісія**, що пов'язані із лінією цінової пропозиції або сервісною роботою за договором, має бути унікальною. Якщо ж комбінація проекту та класу транзакції пов'язується з кількома сервісними лініями або рядками цінової пропозиції, то програма Project Operations не працюватиме належним чином.

## <a name="billing-types"></a>Типи виставлення рахунків

Поле **Тип виставлення рахунка** визначає концепцію оплатності. Це набір параметрів з такими можливими значеннями:

- **Оплатна**: вартість, яка розраховується цією роллю/категорією — це пряма вартість, від якої залежить виконання проекту, а клієнт буде платити за цю роботу. Платіж може бути введений як час і матеріал або фіксована ціна домовленості. Однак працівник, який витрачає цей час, отримає відповідний кредит за оплачуване використання свого часу.
- **Неоплатна**: вартість, яка нараховується на цю роль/категорію, вважається прямою вартістю, від якої залежить виконання проекту, навіть якщо клієнт не визнає цей факт і не буде платити за цю роботу. Працівник, який витрачає цей час, не отримує кредит за його платне використання.
- **Додатково**: вартість, яка нараховується на цю роль/категорію, вважається прямою вартістю, від якої залежить виконання проекту, і клієнт визнає цей факт. Працівник, який витрачає цей час, отримує кредит за його платне використання. Однак ця вартість не виставляється клієнту.
- **Недоступні**: витрати, що виникли на внутрішніх проектах, які не потребують відстеження прибутку, але відстежуються за допомогою цього параметра.

## <a name="invoice-schedule"></a>Розклад виставлення рахунків

Розклад рахунків — це низка дат, коли відбувається створення рахунка-фактури для проекту. Крім того, можна створити розклад рахунка-фактури в рядку цінової пропозиції. Кожен рядок цінової пропозиції може мати власний розклад для рахунка-фактури. Щоб створити розклад рахунка-фактури, необхідно вказати зазначені нижче значення атрибутів.

- Дата початку виставлення рахунків 
- Дата доставки, яка відповідає даті завершення виставлення рахунків у проекті
- Частота виставлення рахунків

Ці три значення атрибутів використовуються для створення попереднього набору дат для видання рахунків-фактур.

## <a name="invoice-frequency"></a>Частота виставлення рахунків

Частота рахунка-фактури — це сутність, в якій зберігаються значення атрибутів, які дають змогу виразити частоту створення рахунків-фактур. Нижче наведено атрибути, які виражають або визначають сутність частоти рахунка.

- **Період**: підтримуються періоди щомісяця, раз на два тижні, щотижня. 
- **Запусків за період**: для тижневих та двотижневих періодів можна визначати лише один запуск за період. Для щомісячних періодів можна визначати між одним і чотирма запусками за період. 
- **Дні запуску**: це дні, коли має виконуватися виставлення рахунка-фактури. Ви можете настроїти цей атрибут двома способами:
  - **У будні дні**: наприклад, можна вказати, що виставлення рахунка-фактури виконується щопонеділка або кожного другого понеділка. Клієнти, яким треба настроїти виставлення рахунка в робочий день, можуть надавати перевагу такій конфігурації. 
  - **Дні календаря**: наприклад, можна вказати, що виставляння рахунка-фактури запускається на сьомий та двадцять перший дні кожного місяця. Деякі організації можуть віддати перевагу цьому типу конфігурації, оскільки це допомагає гарантувати, що виставляння рахунка-фактури виконується щомісяця за фіксованим розкладом.
  
### <a name="invoice-schedule-for-a-fixed-price-quote-line"></a>Розклад рахунка для лінії з фіксованою ціною у пропозиції

Для лінії з фіксованою пропонованою ціною можна використати сітку **Розклад рахунка**, щоб створити проміжні етапи виставлення рахунків, які дорівнюють значенню в цій лінії цінової пропозиції.

- Щоб створити рівномірно розділені проміжні етапи виставлення рахунка, виберіть частоту рахунка-фактури, введіть дату початку виставлення рахунків в рядку цінової пропозиції та виберіть **Дата запитаного завершення** для цінової пропозиції в розділі **Зведення** заголовка цінової пропозиції. Потім виберіть **Створити періодичних проміжних етапів**, щоб створити однаково розподілені етапи на основі вибраної частоти рахунка-фактури. 
- Щоб створити проміжний етап для виставлення рахунка на загальну суму, створіть проміжний етап, а потім введіть значення рядка цінової пропозиції як суми проміжного етапу.
- Щоб створити проміжні етапи виставлення рахунків, які базуються на певних завданнях плану проекту, створіть проміжний етап і зіставте його з елементом розкладу проекту в інтерфейсі проміжного рахунка.
