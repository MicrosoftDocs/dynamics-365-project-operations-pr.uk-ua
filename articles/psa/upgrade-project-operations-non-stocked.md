# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Перехід від автоматизації служби проектів до операцій із проектами

Ми раді оголосити про перший з трьох етапів оновлення від Microsoft Dynamics 365 Project Service Automation до Dynamics 365 Project Operations. Ця стаття надає огляд для клієнтів, які вирушають у цю захоплюючу подорож. Майбутні статті включатимуть міркування розробників та детальну інформацію про покращення функцій. Вони не лише нададуть вказівки, які допоможуть вам підготуватися до оновлення до Project Operations, але й пояснять, чого ви можете очікувати після оновлення.

Програма доставки оновлення буде розділена на три етапи.

| Оновлення доставки | Фаза 1 (січень 2022 року) | Фаза 2 (Квітнева хвиля 2022) | Фаза 3  |
|------------------|------------------------|---------------------------|---------------------------|
| Відсутність залежності від ієрархічної структури робіт (WBS) для проектів | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| WBS в рамках поточних підтримуваних обмежень діяльності проекту | | :heavy_check_mark: | :heavy_check_mark: |
| WBS виходить за межі поточних підтримуваних операцій Project, включаючи підтримку клієнта Project для настільних комп’ютерів | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>Функції процесу оновлення 

У рамках процесу оновлення ми додали журнали оновлення до карти сайту, щоб адміністратори могли легше діагностувати збої. На додаток до нового інтерфейсу будуть додані нові правила перевірки для забезпечення цілісності даних після оновлення. Наступні перевірки будуть додані до процесу оновлення.

| Перевірки | Фаза 1 (січень 2022 року) | Фаза 2 (Квітнева хвиля 2022) | Фаза 3  |
|-------------|------------------------|---------------------------|---------------------------|
| WBS буде перевірено на наявність поширених порушень цілісності даних (наприклад, призначень ресурсів, які пов’язані з одним батьківським завданням, але мають різні батьківські проекти). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено відповідно до [відомих обмежень Project for the Web](/project-for-the-web/project-for-the-web-limits-and-boundaries). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено відповідно до відомих обмежень клієнта Project для настільних комп’ютерів. | |  | :heavy_check_mark: |
| Книжкові ресурси та календарі проектів оцінюватимуться за поширеними несумісними календарними правилами. | | :heavy_check_mark: | :heavy_check_mark: |

На етапі 2 клієнти, які переходять до project operations, оновлять свої існуючі проекти до можливостей лише для читання для планування проекту. У цьому досвіді лише для читання повний WBS буде видно в сітці відстеження. Щоб відредагувати WBS, менеджери проектів можуть вибрати **Конвертувати** на головній **сторінці проектів**. Після цього фоновий процес оновить проект, щоб він підтримував нові можливості планування проекту з веб-програми Project для Інтернету. Цей етап підходить для клієнтів, які мають проекти, які вписуються у [відомі межі Project for the Web](/project-for-the-web/project-for-the-web-limits-and-boundaries).

На етапі 3 буде додано підтримку клієнта Project для настільних комп’ютерів на користь клієнтів, які хочуть продовжувати редагувати свої проекти з цієї програми. Однак, якщо наявні проекти буде перетворено на новий Project для веб-інтерфейсу, доступ до надбудови буде вимкнуто для кожного перетвореного проекту.

## <a name="prerequisites"></a>вимоги

Щоб мати право на оновлення фази 1, клієнт має відповідати таким критеріям:

- Цільове середовище не повинно містити жодних записів у **msdyn_projecttask** суб’єкта господарювання.
- Дійсні ліцензії на операції з проектами мають бути призначені всім активним користувачам клієнта. 
- Замовник повинен перевірити процес оновлення принаймні в одному невиробничому середовищі, яке має репрезентативний набір даних, узгоджений з виробничими даними.
- Цільове середовище має бути оновлено до випуску оновлень служби project service 41 (3.10.62.162) або новішої версії.

Передумови для фази 2 і фази 3 будуть оновлюватися з наближенням загальних дат доступності.

## <a name="licensing"></a>Ліцензування

Якщо у вас є активні ліцензії на автоматизацію служби проектів, можна інсталювати та використовувати операції project, що включає всі можливості автоматизації служби проектування тощо. Таким чином, ви можете протестувати можливості Project Operations, продовжуючи використовувати автоматизацію служби проектів у виробництві. Після закінчення терміну дії ліцензій на автоматизацію служби проектування вам доведеться перейти до Project Operations. Плануючи цей перехід, потрібно враховувати той факт, що ліцензія на операції з проектом не містить ліцензію на автоматизацію служби проектів.

## <a name="testing-and-refactoring-customizations"></a>Налаштування тестування та рефакторингу

Як відправна точка, імпортуйте всі налаштування в чисте середовище Project Operations (Lite), щоб підтвердити, що імпорт успішний, і що бізнес-операції поводяться так, як очікувалося.

Ось на що слід звернути увагу:

- Імпорт може не вдатися через відсутність залежностей. Іншими словами, поля посилань настройок або інші компоненти, видалені під час операцій project. В цьому випадку видаліть ці залежності з середовища розробки.
- Якщо ваші некеровані та керовані рішення включають компоненти, які не настроєно, видаліть ці компоненти з рішення. Наприклад, коли ви настроюєте **сутність проекту**, додайте до свого рішення лише заголовок сутності. Не додавайте всі поля. Якщо ви раніше додавали всі субкомпоненти, можливо, доведеться вручну створити нове рішення та додати до нього відповідні компоненти.
- Форми та подання можуть відображатися не так, як очікувалося. За певних обставин, якщо ви настроїли будь-яку з готових форм або подань, настройки можуть запобігти набуттю чинності нових оновлень у project operations. Щоб виявити ці проблеми, радимо паралельно переглянути чисту інсталяцію "Операції проекту" та інсталяцію "Операції проекту", яка містить настроювання. Порівняйте найбільш часто використовувані форми у вашому бізнесі, щоб підтвердити, що ваша версія форми все ще має сенс і не пропускає щось у чистій версії форми. Виконайте перевірку поруч для всіх настроїв.
- Бізнес-логіка може зазнати невдачі під час виконання. Оскільки посилання на поля в компонентах plug-in не перевіряються під час імпорту, бізнес-логіка може вийти з ладу через посилання на поля, які більше не існують, і ви можете отримати повідомлення про помилку, яке нагадує такий приклад: сутність "Project" не містить атрибута з Name = "msdyn_plannedhours" та NameMapping = "Logical"." У цьому випадку змініть налаштування, щоб вони використовували нові поля. Якщо ви використовуєте автоматично згенеровані класи проксі-серверів і сильні посилання на типи у своїй логіці плагіна, подумайте про те, щоб відновити ці проксі-сервери з чистої інсталяції. Таким чином, ви можете легко визначити всі місця, де ваші зовнішні модулі залежать від застарілих полів.

Після оновлення настройок для чистого імпорту операцій project перейдіть до наступних кроків.

## <a name="end-to-end-testing-in-development-environments"></a>Наскрізне тестування в середовищах розробки

### <a name="initiate-upgrade"></a>Ініціювати оновлення 

1. Power Platform У Центрі адміністрування знайдіть і виберіть своє середовище. Потім в додатках знайдіть і виберіть **Dynamics 365 Project Operations**.
2. Натисніть кнопку **Інсталювати**, щоб розпочати оновлення. Центр Power Platform адміністрування представить цю інсталяцію як нову інсталяцію. Однак наявність більш ранньої версії Project Service Automation буде виявлено, а існуюча установка буде модернізована.

    Після завершення оновлення в середовищі має відображатися, що project Operations інстальовано, а автоматизацію служби project не інстальовано.

    > [!NOTE]
    > Залежно від обсягу даних у середовищі, оновлення може зайняти кілька годин. Основна команда, яка керує оновленням, повинна відповідно планувати та запускати оновлення в неробочий час. У деяких випадках, якщо обсяг даних великий, оновлення слід запустити протягом вихідних. Рішення про планування має ґрунтуватися на результатах тестування в нижчих середовищах.

3. За потреби оновлюйте спеціальні рішення. На цьому етапі розгорніть усі зміни, внесені до настройок, у [розділі Настроювання тестування та рефакторингу](#testing-and-refactoring-customizations) цієї статті.
4. Перейдіть до **розділу Рішення для** настройок \>**і** виберіть, щоб видалити **рішення "Застарілі компоненти**" для операцій проекту.

    Це рішення є тимчасовим рішенням, яке містить існуючу модель даних та компоненти, які присутні під час оновлення. Видаляючи цей розчин, ви видаляєте всі поля і компоненти, які більше не використовуються. Таким чином, ви допомагаєте спростити інтерфейс і полегшити інтеграцію та розширення.
    
### <a name="validate-common-scenarios"></a>Перевірка поширених сценаріїв

Під час перевірки конкретних настройок рекомендуємо також переглянути бізнес-процеси, які підтримуються в програмах. Ці бізнес-процеси включають, але не обмежуються ними, створення суб’єктів продажу, таких як котирування та контракти, а також створення проектів, які включають WBS та затвердження фактичних даних.

## <a name="major-changes-between-project-service-automation-and-project-operations"></a>Основні зміни між автоматизацією обслуговування проектів та операціями з проектами

### <a name="project-planning"></a>Планування проектів

Можливості планування проекту в Project Operations більше не залежать від комбінованої логіки на стороні клієнта та логіки на стороні сервера. Натомість Project Operations використовує Project для Інтернету як механізм планування. Ця зміна можливостей планування дає змогу використовувати кілька нових функцій, таких як подання дошки та діаграми Ганта, планування на основі ресурсів, [елементи контрольного списку завдань](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c) і режими планування проекту. Нові можливості планування також підтримуються багатим набором нових [інтерфейсів прикладного програмування (API).](../project-management/schedule-api-preview.md) Ці api призначені для забезпечення того, щоб жодна програмна операція зі створення, оновлення або видалення сутності в СДР не пошкодила обчислювані поля в розкладі.

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>Як інсталювати операції project до того, як засіб оновлення стане доступним?

Існує два варіанти інсталяції project operations до того, як засіб оновлення стане доступним:

- Забезпечення нового середовища.
- Розгортайте операції project окремо в будь-якій організації з продажу, де немає функції автоматизації служби проектів.

> [!NOTE]
> Якщо автоматизацію служби project інстальовано в організації, але вона не використовувалася, її можна видалити. Після повного видалення функції автоматизації служби проекту операції project можна інсталювати в тій самій організації.
