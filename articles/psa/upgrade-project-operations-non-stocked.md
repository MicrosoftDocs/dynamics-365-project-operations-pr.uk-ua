# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Оновлення від автоматизації служб проекту до операцій проекту

Ми раді оголосити про перший з трьох етапів оновлення від Microsoft Dynamics 365 Project Service Automation до Dynamics 365 Project Operations. Ця тема містить огляд для клієнтів, які приступають до цієї захоплюючої подорожі. Майбутні теми включатимуть міркування розробників та детальну інформацію про покращення функцій. Вони не тільки нададуть вказівки, які допоможуть вам підготуватися до оновлення до операцій із проектами, але й пояснять, чого можна очікувати після оновлення.

Програма доставки оновлення буде розділена на три етапи.

| Оновити доставку | Етап 1 (січень 2022 р.) | Фаза 2 (квітнева хвиля 2022) | Етап 3  |
|------------------|------------------------|---------------------------|---------------------------|
| Відсутність залежності від структури розбивки роботи (WBS) для проектів | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| WBS в межах поточних підтримуваних меж операцій проекту | | :heavy_check_mark: | :heavy_check_mark: |
| WBS за межами поточного підтримуваних меж операцій project, включаючи підтримку клієнта робочого стола Project | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>Функції процесу оновлення 

В рамках процесу оновлення ми додали журнали оновлення на карту сайту, щоб адміністратори могли легше діагностувати збої. На додаток до нового інтерфейсу, нові правила перевірки будуть додані для забезпечення цілісності даних після оновлення. До процесу оновлення буде додано такі перевірки.

| Перевірки | Етап 1 (січень 2022 р.) | Фаза 2 (квітнева хвиля 2022) | Етап 3  |
|-------------|------------------------|---------------------------|---------------------------|
| WBS буде перевірено на загальні порушення цілісності даних (наприклад, призначення ресурсів, які пов'язані з одним батьківським завданням, але мають різні батьківські проекти). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено відповідно до [відомих обмежень проекту для Інтернету](/project-for-the-web/project-for-the-web-limits-and-boundaries). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено відповідно до відомих обмежень клієнта робочого стола проекту. | |  | :heavy_check_mark: |
| Забронювані ресурси та календарі проектів оцінюватимуться за загальними несумісними винятками правил календаря. | | :heavy_check_mark: | :heavy_check_mark: |

На етапі 2 клієнти, які переходять на project operations, матимуть свої існуючі проекти, оновлені до досвіду лише для читання для планування проекту. У цьому досвіді лише для читання повний WBS буде видно в сітці відстеження. Щоб змінити WBS, керівники проектів можуть вибрати перетворити **на** головній **сторінці проектів**. Потім фоновий процес оновить проект таким чином, щоб він підтримував новий досвід планування проектів із програми Project for the Web. Цей етап підходить для клієнтів, які мають проекти, які вписуються в [відомі межі проекту для Інтернету](/project-for-the-web/project-for-the-web-limits-and-boundaries).

На етапі 3 буде додано підтримку настільного клієнта Project на користь клієнтів, які хочуть продовжувати редагувати свої проекти з цієї програми. Однак, якщо наявні проекти буде перетворено на новий проект для веб-досвіду, доступ до надбудови буде вимкнуто для кожного перетвореного проекту.

## <a name="prerequisites"></a>вимоги

Щоб мати право на оновлення фази 1, клієнт повинен відповідати таким критеріям:

- Цільове середовище не повинно містити записів у **msdyn_projecttask** сутності.
- Дійсні ліцензії на операції з проектом повинні призначатися всім активним користувачам клієнта. 
- Клієнт повинен перевірити процес оновлення принаймні в одному невиробничому середовищі, яке має репрезентативний набір даних, який узгоджується з виробничими даними.
- Цільове середовище має бути оновлено до випуску оновлень автоматизації служби Project 41 (3.10.62.162) або пізнішої версії.

Передумови для фази 2 та фази 3 будуть оновлюватися з наближенням загальних дат доступності.

## <a name="licensing"></a>Ліцензування

Якщо у вас є активні ліцензії на автоматизацію служб Project, можна інсталювати та використовувати операції project, які включають всі можливості автоматизації служб project тощо. Таким чином, ви можете перевірити можливості операцій проекту, продовжуючи використовувати автоматизацію служби project у виробництві. Після закінчення терміну дії ліцензій на автоматизацію служб project доведеться перейти до операцій проекту. Під Вільний час планування цього переходу, ви повинні враховувати той факт, що ліцензія операції з проектами не містить ліцензії на автоматизацію служби project.

## <a name="testing-and-refactoring-customizations"></a>Тестування та рефакторинг настроювань

Як відправна точка, імпортувати всі налаштування в чистому середовищі операцій проекту (lite), щоб підтвердити, що імпорт є успішним, і що бізнес-операції поводяться належним чином.

Ось деякі речі, на які слід звернути увагу:

- Імпортування може не вдатися через відсутність залежностей. Іншими словами, настроювання посилаються на поля або інші компоненти, які були видалені в операціях project. У цьому випадку видаліть ці залежності з середовища розробки.
- Якщо некеровані та керовані рішення містять компоненти, які не настроєно, видаліть ці компоненти з рішення. Наприклад, під час **настроювання сутності Проекту** додайте до рішення лише заголовок сутності. Не додавайте всі поля. Якщо ви раніше додавали всі підкомпоненти, можливо, доведеться вручну створити нове рішення та додати до нього відповідні компоненти.
- Форми та подання можуть відображатися не належним чином. За певних обставин, якщо ви налаштували будь-яку з нестандартних форм або подань, настроювання можуть перешкоджати введенню нових оновлень у операціях project. Щоб визначити ці проблеми, рекомендовано виконати паралельний огляд чистої інсталяції операцій із проектом і інсталяції операцій із проектом, яка містить настроювання. Порівняйте найбільш часто використовувані форми у вашій компанії, щоб підтвердити, що ваша версія форми все ще має сенс і чогось не втрачає в чистій версії форми. Виконайте однаковий тип паралельного огляду для будь-яких подань, які ви налаштували.
- Бізнес-логіка може зазнати невдачі під час виконання. Оскільки посилання на поля у компонентах plug-in не перевіряються під час імпортування, бізнес-логіка може не працювати через посилання на поля, які більше не існують, і може з'явитися повідомлення про помилку приблизно такого прикладу: сутність "Project" не містить атрибута з іменем = "msdyn_plannedhours" та NameMapping = "Логічний". У цьому випадку змініть налаштування таким чином, щоб вони використовували нові поля. Якщо в логіці плагіна використовуються автоматично згенеровані проксі-класи та посилання на сильний тип, подумайте про відновлення цих проксі-серверів від чистої інсталяції. Таким чином, ви можете легко визначити всі місця, де ваші плагіни залежать від застарілих полів.

Після оновлення настройок для чистого імпорту операцій project, перейдіть до наступних кроків.

## <a name="end-to-end-testing-in-development-environments"></a>Наскрізне тестування в середовищах розробки

### <a name="initiate-upgrade"></a>Ініціювання оновлення 

1. Power Platform У Центрі адміністрування знайдіть і виберіть своє оточення. Потім у програмах знайдіть і виберіть **Dynamics 365 Project Operations**.
2. Натисніть кнопку **Інсталювати**, щоб розпочати оновлення. Центр Power Platform адміністрування представить цю інсталяцію як нову інсталяцію. Однак буде виявлено наявність попередньої версії project service automation, а існуючу інсталяцію буде оновлено.

    Після завершення оновлення середовище має показати, що операції з проектом інстальовано, і що автоматизація служби project не інстальовано.

    > [!NOTE]
    > Залежно від обсягу даних у навколишньому середовищі, оновлення може зайняти кілька годин. Основна команда, яка керує оновленням, повинна планувати відповідно та запускати оновлення в неробочі години. У деяких випадках, якщо обсяг даних великий, оновлення слід запустити протягом вихідних. Рішення про планування має ґрунтуватися на результатах тестування в нижчих середовищах.

3. Оновіть власні рішення відповідно. На цьому етапі розгорніть усі зміни, внесені до настройок, у [розділі Тестування та рефакторинг настроювань](#testing-and-refactoring-customizations) цього розділу.
4. Перейдіть до розділу **Рішення для параметрів** \> **і** виберіть, щоб видалити **рішення для застарілих компонентів** операцій проекту.

    Це рішення є тимчасовим рішенням, яке містить існуючу модель даних і компоненти, які присутні під час оновлення. Видаливши це рішення, видаліть усі поля та компоненти, які більше не використовуються. Таким чином, ви допомагаєте спростити інтерфейс і полегшити інтеграцію та розширення.
    
### <a name="validate-common-scenarios"></a>Перевірка поширених сценаріїв

Під час перевірки певних настройок рекомендовано також переглянути бізнес-процеси, які підтримуються в різних програмах. Ці бізнес-процеси включають, але не обмежуються ними, створення об'єктів збуту, таких як котирування та контракти, а також створення проектів, які включають WBS та затвердження фактичних даних.

### <a name="project-planning"></a>Планування проектів

Можливості планування проекту в project operations більше не покладаються на комбіновану логіку на стороні клієнта та логіку на стороні сервера. Натомість програма Project Operations використовує програму Project для Інтернету як механізм планування. Ця зміна можливостей планування дає змогу виконувати кілька нових функцій, таких як подання Дошки та Ганта, планування на основі ресурсів, [елементи](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c) контрольного списку завдань і режими планування проектів. Нові можливості планування також підтримуються багатим набором нових [інтерфейсів прикладного програмування (API)](../project-management/schedule-api-preview.md). Ці API призначені для забезпечення того, щоб гарантувати, що жодна програмна операція для створення, оновлення або видалення сутності в WBS не псує обчислювані поля в розкладі.

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>Як інсталювати операції project, перш ніж стане доступним інструментарій оновлення?

Існує два варіанти інсталяції операцій project, перш ніж доступний інструментарій оновлення:

- Створення нового середовища.
- Розгортайте операції проекту окремо в будь-якій організації збуту, де немає автоматизації служби project.

> [!NOTE]
> Якщо автоматизація служби Project інстальовано в організації, але вона не використовувалася, її можна видалити. Після повного видалення автоматизації служби project, операції з проектами можна інсталювати в тій же організації.
