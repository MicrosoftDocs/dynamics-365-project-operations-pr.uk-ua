# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Оновлення від автоматизації служби проектів до операцій проекту

Ми раді оголосити про перший з трьох етапів оновлення від Microsoft Dynamics 365 Project Service Automation до Dynamics 365 Project Operations. Ця тема містить огляд для клієнтів, які приступають до цієї захоплюючої подорожі. Майбутні теми включатимуть міркування для розробників і докладну інформацію про покращення функцій. Вони не тільки нададуть інструкції, які допоможуть вам підготуватися до оновлення до операцій проекту, але й пояснять, що можна очікувати після оновлення.

Програма доставки оновлення буде розділена на три етапи.

| Оновлення доставки | Етап 1 (січень 2022) | Фаза 2 (Квітнева хвиля 2022) | Фаза 3 (Квітнева хвиля 2022) |
|------------------|------------------------|---------------------------|---------------------------|
| Відсутність залежності від структури розбивки роботи (WBS) для проектів | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| WBS в рамках в даний час підтримується в рамках проектних операцій | | :heavy_check_mark: | :heavy_check_mark: |
| WBS за межами поточних підтримуваних обмежень операцій проекту, включаючи підтримку клієнта робочого стола Project | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>Функції процесу оновлення 

У рамках процесу оновлення ми додали журнали оновлення до карти сайту, щоб адміністратори могли легше діагностувати помилки. На додаток до нового інтерфейсу, нові правила перевірки будуть додані для забезпечення цілісності даних після оновлення. До процесу оновлення буде додано такі перевірки.

| Перевірки | Етап 1 (січень 2022) | Фаза 2 (Квітнева хвиля 2022) | Фаза 3 (Квітнева хвиля 2022) |
|-------------|------------------------|---------------------------|---------------------------|
| WBS буде перевірено на загальні порушення цілісності даних (наприклад, призначення ресурсів, пов'язані з одним батьківським завданням, але мають різні батьківські проекти). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено на [відомих обмеженнях проекту для Інтернету](/project-for-the-web/project-for-the-web-limits-and-boundaries). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено на відомих обмеження клієнта робочого стола Project. | | :heavy_check_mark: | :heavy_check_mark: |
| Заброньовані ресурси та календарі проектів буде оцінено на загальних несумісних винятках правил календаря. | | :heavy_check_mark: | :heavy_check_mark: |

На етапі 2 клієнти, які оновлять до операцій проекту, оновлять свої існуючі проекти до досвіду лише для читання для планування проекту. У цьому досвіді лише для читання, повна WBS буде видно в сітці відстеження. Щоб змінити WBS, менеджери проектів можуть вибрати **Перетворити** на головній сторінці **проектів**. Фоновий процес оновить проект таким чином, щоб він підтримував новий досвід планування проекту з project for the Web. Цей етап підходить для клієнтів, які мають проекти, які відповідають [відомим межам проекту для](/project-for-the-web/project-for-the-web-limits-and-boundaries) Інтернету.

На етапі 3 буде додано підтримку клієнта робочого стола Project на користь клієнтів, які хочуть продовжувати редагувати свої проекти з цієї програми. Однак, якщо наявні проекти перетворюються на новий проект для веб-досвіду, доступ до надбудови буде вимкнуто для кожного перетвореного проекту.

## <a name="prerequisites"></a>вимоги

Щоб мати право на оновлення фази 1, клієнт повинен відповідати таким критеріям:

- Цільове середовище не повинно містити записів у **сутності** msdyn_projecttask.
- Дійсні ліцензії на операції з проектами повинні бути призначені всім активним користувачам клієнта. 
- Клієнт повинен перевірити процес оновлення принаймні в одному невиробничому середовищі, яке має репрезентативний набір даних, узгоджений із виробничими даними.
- Цільове середовище має бути оновлено до оновлення служби Project Service Оновлень випуск 38 або пізнішої версії.

Передумови для фази 2 і фази 3 будуть оновлені в міру наближення загальних дат доступності.

## <a name="licensing"></a>Ліцензування

Якщо у вас є активні ліцензії на автоматизацію служби Project, можна інсталювати та використовувати операції проекту, які включають в себе всі можливості автоматизації служби project та багато іншого. Таким чином, ви можете перевірити можливості проект операцій, поки ви продовжуєте використовувати автоматизацію служби project у виробництві. Після закінчення терміну дії ліцензій на автоматизацію служби project project доведеться перейти до операцій проекту. Під Вільний час планування цього переходу, потрібно враховувати той факт, що ліцензія project operations не містить ліцензії автоматизації служби project.

## <a name="testing-and-refactoring-customizations"></a>Тестування та рефакторинг настроювань

Як відправна точка імпортуйте всі настроювання в чисте середовище операцій проекту (lite), щоб підтвердити успішний імпорт, і що бізнес-операції поводяться належним чином.

Ось деякі речі, на які слід звернутись:

- Імпорт може не вдатися через відсутність залежностей. Іншими словами, настроювання посилання поля або інші компоненти, які були видалені в операціях проекту. У цьому випадку видалити ці залежності з середовища розробки.
- Якщо некеровані та керовані рішення містять компоненти, які не настроєно, видаліть ці компоненти з рішення. Наприклад, під час настроювання **сутності Project** додайте до рішення лише заголовок сутності. Не додавайте всі поля. Якщо ви раніше додали всі підкомпоненти, можливо, доведеться вручну створити нове рішення та додати до нього відповідні компоненти.
- Форми та подання можуть не відображатися як неочікувані. У деяких випадках, якщо ви налаштували будь-яку з нестандартних форм або подань, настроювання можуть перешкодити вступу в силу нових оновлень в операціях Проекту. Щоб визначити ці проблеми, рекомендовано виконати паралельний огляд чистої інсталяції операцій проекту та інсталяції операцій проекту, яка включає настроювання. Порівняйте найпоширеніші форми у вашій компанії, щоб підтвердити, що ваша версія форми все ще має сенс і не пропускає щось з чистої версії форми. Виконайте той самий тип паралельної перевірки для будь-яких подань, які ви налаштували.
- Бізнес-логіка може не вдатися під час виконання. Оскільки посилання на поля в компонентах plug-in не перевіряються під час імпортування, бізнес-логіка може не вдатися через посилання на поля, яких більше не існує, і може з'явитися повідомлення про помилку приблизно такого вигляду: "Сутність "Проект" не містить атрибут з іменем = "msdyn_plannedhours" та NameMapping = "Logical"." У цьому випадку змініть настроювання таким чином, щоб вони використовували нові поля. Якщо ви використовуєте автоматично згенеровані проксі-класи та сильні посилання на тип у вашій логіці плагіна, розгляньте можливість регенерації цих проксі-серверів з чистої інсталяції. Таким чином, ви можете легко визначити всі місця, де ваші плагіни залежать від застарілих полів.

Після оновлення настройок для чистого імпорту операцій проекту перейдіть до наступних кроків.

## <a name="end-to-end-testing-in-lower-environments"></a>Наскрізне тестування в нижчих середовищах

### <a name="run-the-upgrade-in-production"></a>Запустити оновлення у виробництві

1. У Power Platform Центрі адміністрування знайдіть і виберіть своє середовище. Потім у програмах знайдіть і виберіть **Dynamics 365 Project Operations**.
2. Натисніть **кнопку** Інсталювати, щоб розпочати оновлення. Power Platform Центр адміністрування представить цю інсталяцію як нову інсталяцію. Однак, наявність попередньої версії автоматизації служби project буде виявлено, і наявна інсталяція буде оновлено.

    Після завершення оновлення середовище має показати, що інстальовано операції проекту, а також не інстальовано автоматизацію служби Project.

    > [!NOTE]
    > Залежно від обсягу даних у середовищі, оновлення може зайняти кілька годин. Основна команда, яка керує оновленням, повинна планувати відповідно та запускати оновлення в неробочий час. У деяких випадках, якщо обсяг даних великий, оновлення слід запустити протягом вихідних. Рішення про планування має ґрунтуватися на результатах тестування в нижчих середовищах.

3. Оновіть власні рішення відповідно. На цьому етапі розгорніть будь-які зміни, внесені до настройок у [розділі Тестування та рефакторинг настроювань](#testing-and-refactoring-customizations) цього розділу.
4. Перейдіть до **настройки** \> **рішення** та виберіть, щоб видалити **рішення проект операцій застарілих** компонентів.

    Це рішення є тимчасовим рішенням, яке містить наявну модель даних і компоненти, присутні під час оновлення. Видаливши це рішення, видалити всі поля та компоненти, які більше не використовуються. Таким чином, ви допомагаєте спростити інтерфейс і полегшити інтеграцію та розширення.

## <a name="major-changes-between-project-service-automation-and-project-operations"></a>Основні зміни між автоматизацією проектних послуг та операціями проектів

У цьому розділі наведено короткий опис основних змін, які можна очікувати між автоматизацією служби project та операціями проекту.

### <a name="project-planning"></a>Планування проектів

Можливості планування проекту в операціях проекту більше не покладаються на комбінацію логіки на стороні клієнта та логіки на стороні сервера. Замість цього, проект операцій використовує проект для Інтернету, як це планування обробника. Ця зміна можливостей планування дає змогу кілька нових функцій, таких як подання дошки та ганта, планування на основі ресурсів, [елементи контрольного списку завдань](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c) і режими планування проекту. Нові можливості планування також підтримуються багатим набором нових [інтерфейсів прикладного програмування (API).](../project-management/schedule-api-preview.md) Ці API призначені для забезпечення того, щоб жодна програмна операція для створення, оновлення або видалення сутності в WBS не було пошкоджено обчислювані поля в розкладі.

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>Як інсталювати операції проекту до того, як буде доступне інструментарій оновлення?

Існує два варіанти інсталяції операцій проекту, перш ніж доступне інструментарій оновлення:

- Забезпечення нового середовища.
- Розгортання операцій проекту окремо в будь-якій організації збуту, де немає автоматизації служби project.

> [!NOTE]
> Якщо в організації інстальовано автоматизацію служби Project, але вона не використовується, її можна видалити. Після повного видалення автоматизації служби project project operations можна інсталювати в одній організації.
