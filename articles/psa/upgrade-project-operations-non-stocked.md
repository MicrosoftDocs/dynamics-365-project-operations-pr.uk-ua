# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Перехід від Project Service Automation до Project Operations

Ми раді представити другий з трьох етапів для оновлення з Microsoft Dynamics 365 Project Service Automation до Microsoft Dynamics 365 Project Operations. У цій статті наведено огляд для клієнтів, які приєднуються до цієї захоплюючої подорожі. 

Програму доставки оновлення буде розділено на три етапи.

| Доставка оновлення | Етап 1 (січень 2022 р.) | Етап 2 (листопад 2022 р.) | Етап 3 (квітень 2023 р.)  |
|------------------|------------------------|---------------------------|---------------------------|
| Відсутність залежності від робочої структури проекту (WBS) для проектів | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| WBS у підтримуваних зараз обмеженнях Project Operations | | :heavy_check_mark: | :heavy_check_mark: |
| WBS знаходиться за межами наразі підтримуваних обмежень Project Operations, включно з підтримкою Project Desktop Client | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>Функції процедури оновлення 

У рамках процесу оновлення додано журнали оновлення до карти сайту, щоб адміністратори могли легше діагностувати помилки. Окрім нового інтерфейсу будуть додані нові правила перевірки, щоб забезпечити цілісність даних після оновлення. До процесу оновлення буде додано перелічені нижче перевірки.

| Перевірки | Етап 1 (січень 2022 р.) | Етап 2 (листопад 2022 р.) | Етап 3  |
|-------------|------------------------|---------------------------|---------------------------|
| WBS буде перевірено на загальні порушення цілісності даних (наприклад, призначення ресурсів, що пов’язані з одним батьківським завданням, але мають різні батьківські проекти). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено за [відомими обмеженнями Project for the Web](/project-for-the-web/project-for-the-web-limits-and-boundaries). | | :heavy_check_mark: | :heavy_check_mark: |
| WBS буде перевірено за відомими обмеженнями Project Desktop Client. | |  | :heavy_check_mark: |
| Плановані ресурси та календарі проекту оцінюватимуться на наявність загальних винятків у несумісних правилах календаря. | | :heavy_check_mark: | :heavy_check_mark: |

На 2-му етапі у клієнтів, які оновили програму до Project Operations, буде оновлено наявні проекти до доступного лише для читання інтерфейсу для планування проектів. У цьому доступному лише для читання інтерфейсі повна версія WBS відображатиметься в сітці відстеження. Для редагування WBS керівники проектів можуть вибрати [**Перетворити**](/PSA-Upgrade-Project-Conversion.md) на головній сторінці проекту. Потім фоновий процес оновлює проект таким чином, щоб він підтримував нові можливості планування проектів в Project for the Web. Цей етап підходить для клієнтів, які мають проекти, що відповідають [відомим обмеженням Project for the Web](/project-for-the-web/project-for-the-web-limits-and-boundaries).

На етапі 3 буде додано підтримку Project Desktop Client, для клієнтів, які хочуть продовжувати редагувати свої проекти в цій програмі. Проте, якщо наявні проекти перетворено для нового інтерфейсу Project for the Web, доступ до надбудови буде вимкнуто для кожного перетвореного проекту.

## <a name="prerequisites"></a>вимоги

Щоб мати право на Етап 1 оновлення, необхідно виконати такі умови:

- Цільове середовище не має містити записів у сутності **msdyn_projecttask**.
- Дійсні ліцензії на Project Operations мають бути призначені всім активним користувачам. 
- Ви маєте перевірити процес оновлення принаймні в одному неробочому середовищі, яке містить репрезентативний набір даних, який узгоджено з робочим середовищем.
- Цільове середовище слід оновити до Project Service Automation оновлений випуск 37 версії V3.10.58.120 або пізнішої.

Щоб мати право на Етап 2 оновлення, необхідно виконати такі умови:

- Дійсні ліцензії на Project Operations мають бути призначені всім активним користувачам. 
- Ви маєте перевірити процес оновлення принаймні в одному неробочому середовищі, яке містить репрезентативний набір даних, який узгоджено з робочим середовищем.
- Цільове середовище слід оновити до Project Service Automation оновлений випуск 37 версії V3.10.58.120 або пізнішої.
- Середовища, які містять завдання (msdyn_projecttask), підтримуються, лише якщо загальна кількість завдань на проект становить 500 або менше.

Вимоги для Етапу 3 буде оновлено ближче до дати загальної доступності.

## <a name="licensing"></a>Ліцензування

Якщо у вас є активні ліцензії для Project Service Automation, можна інсталювати та використовувати програму Project Operations, що включає всі можливості Project Service Automation та багато іншого. Таким чином можна протестувати можливості Project Operations, продовжуючи використовувати Project Service Automation у виробництві. Після завершення терміну дії ліцензій Project Service Automation потрібно буде перейти на Project Operations. Під час планування цього переходу потрібно враховувати той факт, що ліцензія на Project Operations не включає ліцензію на Project Service Automation. Клієнти, які мають сценарії, в яких вони розгорнули Project Service Automation і мають продовжувати використовувати або збільшувати свої ліцензії для PSA, плануючи переходити на Project Operations, можуть створити запит на тимчасові ліцензії на PSA на основі придбаних ліцензій Project Operations. Одна ліцензія на Project Service Automation буде випущена для однієї ліцензії Project Operations. За цим посиланням може запитати тимчасові ліцензії на PSA: aka.ms/ineedpsa

## <a name="testing-and-refactoring-customizations"></a>Налаштування тестування та реструктуризації

Для початку імпортуйте всі настроювання в чисте середовище Project Operations (Lite), щоб переконатися, що імпорт успішно виконано та бізнес-операції виконуються, як очікувалося.

Нижче наведено кілька важливих порад.

- Можливий збій імпорту через відсутність залежностей. Іншими словами, поля посилання на настроювання або інші компоненти, видалені в Project Operations. У цьому разі видаліть ці залежності з середовища розробки.
- Якщо некеровані та керовані рішення містять неналаштовані компоненти, видаліть ці компоненти з рішення. Наприклад, під час налаштування сутності **Проект** додавайте до рішення лише заголовок сутності. Не додавайте всі поля. Якщо раніше було додано всі вкладені компоненти, можливо, знадобиться створити нове рішення вручну та додати до нього відповідні компоненти.
- Форми та подання можуть не відображатися, як очікувалося. У деяких обставинах, якщо ви налаштували будь-яку з готових форм або подань, настроювання можуть не дати новим оновленням у Project Operations набрати чинності. Щоб визначити ці проблеми, ми рекомендуємо виконати паралельну перевірку чистої інсталяції Project Operations з інсталяцією Project Operations, що містить ваші настроювання. Порівняйте форми, які використовуються найчастіше в компанії, щоб переконатися, що версія форми досі є обґрунтованою та в ній немає відсутніх у чистій версії форми компонентів. Виконайте такий самий тип паралельної перевірки для всіх подань, які ви налаштували.
- Бізнес-логіка може не спрацювати під час виконання. Оскільки посилання на поля у ваших компонентах plug-in не перевіряються під час імпорту, бізнес-логіка може дати збій через посилання на поля, які більше не існують, і ви можете отримати повідомлення про помилку, подібне до такого прикладу: «Сутність "Проект" не містить атрибута з Name = 'msdyn_plannedhours' і NameMapping = 'Logical'». У цьому випадку змініть настроювання, щоб вони використовували нові поля. Якщо в логіці компонента plug-in використовуються автоматично створені проксі-класи та посилання на сильний тип, рекомендовано повторно створити ці проксі з чистої інсталяції. Таким чином можна легко визначити всі місця, де компоненти plug-in залежать від вилучених полів.

Після оновлення настроювань для чистого імпорту Project Operations перейдіть до наступних кроків.

## <a name="end-to-end-testing-in-development-environments"></a>Комплексне тестування в середовищах розробки

### <a name="initiate-upgrade"></a>Ініціювати оновлення 

1. В центрі адміністрування Power Platform знайдіть і виберіть ваше середовище. А потім у цих програмах знайдіть і виберіть **Dynamics 365 Project Operations**.
2. Виберіть **Інсталювати**, щоб розпочати оновлення. Центр адміністрування Power Platform представить цю інсталяцію як нову інсталяцію. Проте присутність попередньої версії Project Service Automation буде виявлено, і наявну інсталяцію буде оновлено.

    Після завершення оновлення в середовищі має відображатися, що Project Operations інстальовано, а Project Service Automation не інстальовано.

    Залежно від кількості даних у середовищі оновлення може тривати кілька годин. Основна робоча група, яка керує оновленням, має відповідно запланувати оновлення та запустити оновлення протягом неробочих годин. У деяких випадках, якщо обсяг даних великий, оновлення слід виконувати протягом вихідних. Рішення про планування має базуватись на результатах тестування в нижчих середовищах.

3. За необхідності оновіть настроювані рішення. На цьому етапі розгорніть всі зміни, внесені до настроювань, у розділі [Налаштування тестування та реструктуризації](#testing-and-refactoring-customizations) цієї статті.
4. Перейдіть до сторінки **make.powerapps.com**, у розкривному списку у верхньому правому куті порталу виберіть своє середовище, виберіть **Рішення** в меню ліворуч, виберіть рішення **Вилучені компоненти Project Operations** і **Видалити**.

    Це рішення є тимчасовим рішенням, яке тримає наявну модель даних і компоненти, присутні під час оновлення. Видалення цього рішення призведе до видалення всіх полів і компонентів, які більше не використовуються. Таким чином можна спростити інтерфейс, а також спростити інтеграцію та розширення.
    
### <a name="upgrade-to-project-operations-lite"></a>Оновлення до Project Operations Lite

Нижче наведені кроки, які описують процес оновлення та пов’язаний із ним журнал помилок.

1. **Перевірка версії PSA:** щоб інсталювати Project Operations, потрібно мати версію 3.10.58.120 або вище.
1. **Попередня перевірка:** коли адміністратор ініціює оновлення, система виконує попередню перевірку для кожної сутності, яка є основною для рішення Project Operations. Цей крок перевіряє, чи є всі посилання на сутності дійсними, і це гарантує, що дані, пов’язані з WBS, розташовані в опублікованих обмеженнях Project for the Web.
1. **Оновлення метаданих:** після успішної попередньої перевірки система ініціює зміни схеми та створює вилучене рішення компонентів. Після завершення всієї необхідної реструктуризації налаштувань це застаріле рішення можна видалити. Цей крок є найдовшою частиною процесу оновлення та може тривати до чотирьох годин.
1. **Оновлення даних:** після завершення всіх необхідних змін схеми на кроці оновлення метаданих ваші дані переносяться до нової схеми та виконуються всі необхідні скидання до стандартних значень та перерахування.
1. **Оновлення ядра розкладів проектів:** після успішного оновлення даних назва вкладки **Розклад** на головній сторінці змінюється на **Завдання**. Коли користувач вибирає цю вкладку після оновлення, він спрямовується до сітки відстеження, щоб переглянути версію WBS лише для читання. Щоб відредагувати WBS, вони мають ініціювати [процес перетворення](/PSA-Upgrade-Project-Conversion.md) розкладу. Усі проекти без попереднього встановленого WBS можуть використовувати нові можливості планування безпосередньо без перетворення.
 
### <a name="validate-common-scenarios"></a>Перевірка загальних сценаріїв

Під час перевірки певних налаштувань рекомендовано також перевірити бізнес-процеси, які підтримуються в програмах. Ці бізнес-процеси включають зокрема створення таких сутностей збуту, як цінові пропозиції та сервісні договори, а також створення проектів, які включають WBS і затвердження фактичних даних.

## <a name="major-changes-between-project-service-automation-and-project-operations"></a>Основні відмінності між Project Service Automation і Project Operations

У цьому розділі наведено зведення основних відмінностей, які можна очікувати між Project Service Automation і Project Operations.

### <a name="project-planning"></a>Планування проектів

Можливості планування проектів у Project Operations більше не базуються на логіці клієнта та логіці сервера. Натомість, Project Operations використовує Project for the Web як ядро для планування. Ця зміна можливостей планування дозволяє використовувати кілька нових функцій, наприклад подання «Панель» і «Діаграма Ганта», планування на основі ресурсів, [елементи контрольних списків завдань](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c) і режими планування проектів. Нові можливості планування також підтримуються великим набором нових [інтерфейсів прикладних програм (API)](../project-management/schedule-api-preview.md). Ці API допоможуть гарантувати, що жодні програмні операції зі створення, оновлення або видалення сутності в WBS не пошкоджують в розкладі обчислювані поля.

- [Сервісні договори на основі авансових платежів і платежів на основі абонентського внеску](../pro/sales/set-up-advances-retainer-based-contracts-sales.md)
- [Стан і перевірки граничних обмежень сервісного договору](../pro/proforma-invoicing/manage-nte-status-validations-sales.md)
- Виставлення рахунків на основі завдань

У Project Operations надається додаткова підтримка для панелі Universal Resource Scheduling (URS) і помічника з планування. У хвилі за квітень 2023 року ця нова можливість стане обов’язковою.

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>Як інсталювати Project Operations, перш ніж буде доступним засіб оновлення?

Існує два способи інсталювати Project Operations до того, як засіб оновлення стане доступним.

- Підготовка нового середовища.
- Розгорніть Project Operations окремо в будь-якій організації збуту, в якій немає Project Service Automation.

Якщо Project Service Automation інстальовано в організації, але вона не використовувалася, її можна видалити. Після остаточного видалення Project Service Automation Project Operations можна інсталювати в тій самій організації.
